# 🏗️ 架构总览

EchoSRT 采用**单进程异步架构**，以 FastAPI 为 Web 框架，通过 `asyncio.Queue` 驱动三级流水线，前端使用 React 和 Capacitor 构建，通过 WebSocket 全双工通信接收实时进度。

---

## 层级职责

| 层级 | 路径 | 职责 |
|------|------|------|
| **路由层** | `api/routers/` | 接收 HTTP 请求，参数校验，调用服务层 |
| **服务层** | `api/services/` | 业务逻辑编排：配置持久化、任务分发、工作区操作 |
| **状态层** | `api/state.py` | 全局共享数据结构：三个 `asyncio.Queue` + 两个 Dict |
| **Worker 层** | `api/workers/` | 守护协程，消费队列，驱动 `core/` 引擎 |
| **核心引擎** | `core/` | 纯 Python 模块：FFmpeg、faster-whisper、OpenAI API 翻译 |
| **前端** | `frontend/` | 基于 React 和 Capacitor 构建，通过 WebSocket 全双工通信 |

## 数据流向

```
用户上传视频
    │
    ▼
┌──────────────────┐
│  POST /api/tasks │  ─── 用户指定 steps: ["extract", "transcribe", "translate"]
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│dispatcher_service│  ─── 路径消毒 + 防重复 + 代理测试 → 推入 q_extract
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ worker_extract   │  ─── 消费 q_extract → FFmpeg 提取 audio.wav → 推入 q_transcribe
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ worker_transcribe│  ─── 消费 q_transcribe → 引擎分支 (本地/云端) → 推入 q_translate
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ worker_translate │  ─── 消费 q_translate → LLM API 翻译 → completed
└──────────────────┘

每一步通过 WebSocket (api/ws_manager.py) 向浏览器推送实时进度
```

## 关键设计决策

<details>
<summary><b>1. 单进程多协程 vs 多进程</b></summary>

EchoSRT 选择 **单进程 + asyncio 协程** 而非 Celery/RQ 等多进程方案：

- **CPU 密集操作**（Whisper 推理）通过 `loop.run_in_executor()` 跑在线程池中，释放 GIL 后 Python 线程可并行（C 扩展）
- **I/O 密集操作**（LLM API 调用）天生适合 async/await
- 部署简单：一条 `python app.py` 即可，无需额外的 broker/worker 进程
- 配置热加载：全局 Dict 共享状态，无需序列化到 Redis
</details>

<details>
<summary><b>2. 队列驱动的流水线</b></summary>

三个 `asyncio.Queue` 构成无锁生产者-消费者模型：

```python
# api/state.py
q_extract = asyncio.Queue()      # extract worker → transcribe worker
q_transcribe = asyncio.Queue()   # transcribe worker → translate worker
q_translate = asyncio.Queue()    # translate worker → 完成
```

每个 Worker 是独立的守护协程，通过 `queue.task_done()` 标记完成，支持优雅关闭。
</details>

<details>
<summary><b>3. 基于 React 与 Capacitor 的前端架构</b></summary>

EchoSRT 的前端采用 React 框架构建，并利用 Capacitor 提供跨平台打包能力，不再依赖原生 JS 无构建方案。

好处：组件化开发、更强的数据状态管理、支持跨多端（Web、Android、iOS 等）部署。
</details>

<details>
<summary><b>4. 静态文件服务与 SPA 路由</b></summary>

`app.py` 中通过 `Mount` 将 `/` 路径挂载到 `frontend/` 目录，并注册 SPA 回退路由：

```python
app.mount("/", StaticFiles(directory="frontend", html=True), name="static")
```

`html=True` 使 404 请求回退到 `index.html`，支持前端路由。
</details>

## 启动流程

```
main.py ──启动──▶ uvicorn(app, port=8080)
    │
    ├── 1. 初始化 workspace/ 目录
```
